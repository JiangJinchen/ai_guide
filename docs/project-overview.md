# AI Guide 项目整体介绍

## 1. 项目定位

AI Guide 是一个面向景区导览与运营管理的综合型智能应用项目。它不是单一的聊天机器人，也不是单纯的景点展示站点，而是将游客服务、地图能力、路线规划、内容运营、反馈收集与 AI 对话能力整合到同一套系统中。

当前项目以“灵山胜境”景区为主要业务场景，通过游客端与管理端协同，形成了一个比较完整的智慧景区导览闭环：

- 游客进入首页并查看景区服务
- 根据实时位置或手动选点获取附近景点与服务点
- 通过 AI 对话、景点讲解、活动服务和票务助手快速获取信息
- 基于偏好、时间、位置生成个性化路线
- 查看路线详情并进入路线导航
- 在游玩后沉淀足迹、反馈与行为数据
- 管理端再对内容、运营信息与数据结果进行维护

## 2. 项目目标

项目的核心目标可以拆成三层：

### 2.1 游客体验层

- 降低游客在陌生景区中的决策成本
- 提供更直观的游览路线与内容引导
- 让“问路、找景点、查票务、看活动、去导航”形成连贯体验

### 2.2 景区运营层

- 支撑景区内容维护与运营信息更新
- 收集游客行为与反馈
- 为后续数据分析、服务优化与个性化推荐提供基础

### 2.3 AI 能力层

- 将大模型问答能力和景区知识结合
- 通过 RAG、地图服务与规则逻辑提升回答质量
- 在外部服务不可用时保留可降级的本地体验

## 3. 系统组成

整个项目由四个核心部分组成：

### 3.1 游客端 `frontend/visitor-app`

游客端基于 `Uni-app + Vue 3` 构建，承担主要用户体验。当前主要页面包括：

- `home`：首页服务入口与景区内容概览
- `chat`：AI 数字导游对话
- `guide`：景点讲解列表 / 详情入口
- `route-planning`：路线规划
- `route-detail`：路线详情与顺序编辑
- `route-navigation`：路线导航
- `nearby-spots`：附近景点与服务点
- `ticket-assistant`：票务助手
- `activity-service`：活动服务
- `recommendation`：个性化推荐
- `spot-detail`：单景点详情
- `profile`：足迹、偏好、反馈等个人中心能力

游客端是整个项目里功能最完整、交互最复杂的部分，也是项目展示价值最直接的一侧。

### 3.2 管理端 `frontend/admin-app`

管理端基于 `React + Vite + ECharts`，主要面向内容维护和运营分析。它和游客端不是同一个壳层，而是单独的 Web 应用。

它的典型职责包括：

- 维护景点、票务、活动等基础内容
- 查看用户行为与游客反馈
- 展示统计数据与运营信息

### 3.3 Python 主业务后端 `backend/python`

Python 侧是项目的主业务服务，采用 `FastAPI`。它承担了绝大多数业务 API：

- 游客端业务接口
- 路线规划 / 路线缓存 / 路线分享
- 景点讲解、票务、活动、推荐
- 游客反馈、行为、足迹
- 部分 AI 问答与流式能力接入
- 地图服务调用与缓存

服务入口位于：

- [backend/python/main.py]

核心模块位于：

- `app/api`
- `app/services`
- `app/models`
- `app/utils`

### 3.4 Node.js 辅助服务 `backend/nodejs`

Node.js 侧主要承担辅助型后端能力，技术栈是 `Express + ws + sse`。从依赖和目录结构看，它更偏向：

- 实时通信
- 辅助接口
- 与前端某些实时交互场景的衔接

它不是当前项目的唯一后端，而是 Python 主服务之外的补充层。

## 4. 关键业务能力说明

## 4.1 AI 问答与数字导游

AI 问答不是一个孤立的聊天框，而是游客导览流程中的统一入口。项目通过对话接口把用户自然语言需求连接到具体服务：

- 想知道某个景点的信息
- 想去某个景点
- 想找路线、票务、活动
- 想获取周边服务点

问答结果既可以是文字，也可以进一步转化为服务动作，例如跳转路线规划、导航、景点讲解等。

此外，项目中还包含数字人预览与数字人配置相关能力，为展示型和交互型导览体验留出了空间。

## 4.2 路线规划

路线规划是项目中最有业务价值的模块之一。它综合考虑：

- 游客时间预算
- 游客偏好
- 起点位置
- 必游景点
- 出行方式

后端会生成候选路线，前端则负责展示路线卡片、路线详情和后续的全程导航衔接。

路线相关页面之间的关系大致是：

1. 在 `route-planning` 生成候选路线
2. 在 `route-detail` 查看路线详情、调整顺序、加入景点
3. 在 `route-navigation` 进入实际导航视图

## 4.3 附近景点与位置服务

附近景点页结合定位与地图能力，为用户提供：

- 当前附近景点
- 附近服务点
- 手动切换当前位置
- 快速进入导航

这部分功能既依赖本地位置，也依赖高德地图相关接口。项目中已经实现了：

- 距离缓存
- 地图请求限流
- 本地距离估算兜底

这让它在弱网、限流或外部接口异常时，仍然可以尽量维持可用性。

## 4.4 景点讲解与内容展示

景点讲解能力围绕景区内容资产展开，主要包括：

- 景点基础信息
- 文化介绍
- 亮点说明
- 导览素材资源

这使得系统不是只给“路线”，也能给“内容”。

## 4.5 活动、票务与推荐

这三个模块共同承担游客实际游玩前后的决策支持：

- `activity-service`：查看演出、活动与相关说明
- `ticket-assistant`：查看票务信息、价格、规则、入口
- `recommendation`：根据偏好与场景推荐景点

它们补足了“路线之外”的服务能力，使项目更接近完整景区应用，而不仅是导航工具。

## 4.6 游客画像、足迹与反馈

项目已经具备用户行为与反馈闭环：

- 记录足迹
- 收集反馈
- 保存路线历史
- 管理端后续可基于这些数据做分析

这部分能力为后续运营分析、推荐优化和服务评估提供了数据基础。

## 5. 技术架构概览

## 5.1 前端架构特点

游客端采用 Uni-app，有几个明显特点：

- 适合同时兼顾 H5 / 小程序 / App 方向
- 页面驱动明显，业务流程清楚
- 地图、定位、导航、卡片式服务入口较多

管理端采用 React，承担更标准的运营后台角色。

## 5.2 Python 后端架构特点

Python 主服务具备以下特征：

- 基于 FastAPI 的 REST 风格接口组织
- 使用 SQLAlchemy 进行模型管理
- 在启动阶段会做数据库兼容性补列
- 支持请求日志、422 校验日志与模型预热
- 提供静态资源挂载能力

这说明项目已经从“单接口原型”发展到了“具备基本工程化结构”的阶段。

## 5.3 数据与缓存策略

从代码中能看出项目已经在几个关键点做了工程处理：

- 路线距离缓存
- 路线分享
- 请求级降级逻辑
- RAG 模型预热
- 地图接口限流

这类策略非常重要，因为景区导览类应用对“可用性”和“响应速度”的要求往往高于炫技。

## 7. 项目优势

从整体设计看，这个项目的优势不只是“用了 AI”，而是：

- 把 AI 放进了真实业务流，而不是单独做一个聊天入口
- 同时覆盖游客端与管理端
- 有路线、地图、内容、反馈这些完整场景
- 已经具备一定工程化基础和后续扩展空间
